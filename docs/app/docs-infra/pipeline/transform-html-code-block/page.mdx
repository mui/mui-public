# Transform HTML Code Block

A rehype plugin that transforms `<pre>` elements containing `<code>` blocks into precomputed data for the CodeHighlighter component. This plugin extracts source code from HTML, processes it through syntax highlighting and transformations, then stores the results for efficient client-side rendering.

## Overview

This plugin is typically used in the **second stage** of a markdown-to-HTML processing pipeline, after [`transformMarkdownCode`](../transform-markdown-code/page.mdx) has converted markdown code blocks into HTML structures.

## Key Features

- **Multiple code variants**: Process multiple languages or versions within a single code block
- **Automatic language detection**: Determines file types from `class="language-*"` attributes
- **Custom variant names**: Supports `data-variant` attribute for explicit naming
- **Custom filenames**: Supports `data-filename` attribute for specific file extensions
- **Opt-in transformations**: Code blocks are only transformed when marked with `data-transform="true"`
- **Built-in transformations**: Includes TypeScript-to-JavaScript conversion for marked blocks
- **Syntax highlighting**: Pre-processes code for faster client rendering
- **Lenient emphasis defaults**: Inline `@highlight` and `@focus` regions keep up to 25 lines of surrounding context and up to 60 visible focused lines before collapsing
- **JSX expression cleanup**: Strips trailing semicolons from solo JSX expression lines in `jsx`/`tsx` code blocks
- **Ignore comment stripping**: Automatically removes tool-specific ignore comments (`prettier-ignore`, `eslint-disable`, `@ts-ignore`, `@ts-expect-error`, `@ts-nocheck`) that are noise in documentation
- **Error handling**: Gracefully handles processing errors

## Installation & Usage

```typescript
import { unified } from 'unified';
import rehypeParse from 'rehype-parse';
import transformHtmlCodeBlock from '@mui/internal-docs-infra/pipeline/transformHtmlCodeBlock';

const processor = unified().use(rehypeParse).use(transformHtmlCodeBlock).use(rehypeStringify);
```

## Optional Configuration

You can override the site-wide fallback emphasis window for authored inline code blocks:

```typescript
const processor = unified()
  .use(rehypeParse)
  .use(transformHtmlCodeBlock, {
    paddingFrameMaxSize: 12,
    focusFramesMaxSize: 40,
  })
  .use(rehypeStringify);
```

These values are only the default fallback used by the plugin. Per-block `@padding` and `@min`
directives still take precedence when they are present in the source.

Pass `oversizedFocus: 'hide'` to make blocks whose focused region exceeds `focusFramesMaxSize`
collapse to nothing (an empty collapsed window) instead of showing a truncated slice. See
[`enhanceCodeEmphasis`](../enhance-code-emphasis/page.mdx#collapsing-to-nothing-with-oversizedfocus-hide)
for details.

### Collapse to empty (`collapseToEmpty`)

`collapseToEmpty` is a **render-time** flag that collapses a code block to an _empty_ window: the whole
block is hidden until the reader expands it. Unlike `oversizedFocus: 'hide'` (which only affects
oversized regions and is baked into the precomputed HAST), `collapseToEmpty` never changes the HAST — it
only rewrites how frames render, demoting every collapsed-visible frame (`focus`, `highlighted`,
`padding-*`) to a hidden variant so the existing collapse CSS hides everything. It is orthogonal to
the expanded state.

The three collapse states are: **expanded** (all code), **collapsed** (focus window), and
**collapsed-to-empty** (nothing — this flag).

Set it per block with a fence flag:

````md
```ts collapseToEmpty
console.log('hidden until expanded');
```
````

Or default every authored block to collapse-to-empty via the plugin option, and opt individual
blocks back out with `collapseToEmpty=false`:

```typescript
const processor = unified()
  .use(rehypeParse)
  .use(transformHtmlCodeBlock, { collapseToEmpty: true })
  .use(rehypeStringify);
```

```ts collapseToEmpty=false
// collapses to its focus window instead of an empty window
```

The same flag is available on `CodeHighlighter` (`<CodeHighlighter collapseToEmpty />`) and on demos
created with `createDemo` (`createDemo(url, Component, { collapseToEmpty: true })` or
`<Demo collapseToEmpty />`). A demo factory can default all of its demos to collapse-to-empty with
`collapseToEmpty: true`, and individual demos opt out with `showCollapsedFocus` (meta or
`<Demo showCollapsedFocus />`).

### Start expanded (`initialExpanded`)

`initialExpanded` controls whether a collapsible block starts expanded. Like `collapseToEmpty`, it is
a render-time flag that lives on the content channel (so the loading fallback honors it). Set it per
block (` ```ts initialExpanded ` / ` ```ts initialExpanded=false `), as a `transformHtmlCodeBlock`
option (site-wide default), on `CodeHighlighter` (`<CodeHighlighter initialExpanded />`), or on a
demo (`createDemo(url, Component, { initialExpanded: true })` / `<Demo initialExpanded />`). A demo
factory can default all of its demos to start expanded with `initialExpanded: true`, and individual
demos opt out with `initialCollapsed` (meta or `<Demo initialCollapsed />`).

## Input Examples

### Single Code Block

```html
<pre><code class="language-typescript">
const greeting: string = "Hello, world!";
console.log(greeting);
</code></pre>
```

### Multiple Code Variants

Multi-variant input arrives as the semantic structure that [`transformMarkdownCode`](../transform-markdown-code/page.mdx) emits — a `<section>` of `<figure>` elements, each with a `<figcaption>` label and a `<dl>` holding that variant's files:

```html
<section>
  <figure>
    <figcaption>npm variant</figcaption>
    <dl>
      <dd>
        <pre><code class="language-shell" data-variant="npm">npm install package</code></pre>
      </dd>
    </dl>
  </figure>
  <figure>
    <figcaption>pnpm variant</figcaption>
    <dl>
      <dd>
        <pre><code class="language-shell" data-variant="pnpm">pnpm install package</code></pre>
      </dd>
    </dl>
  </figure>
</section>
```

The variant name comes from the `<figcaption>` text (with the ` variant` suffix stripped), falling back to the first file's `data-variant` attribute. A standalone `<dl>` (without a `<section>` wrapper) is treated as a single variant, with `<dt>` elements naming the files.

### Custom Filenames

```html
<pre><code class="language-javascript" data-filename="app.jsx">
const App = () => <div>Hello React!</div>;
export default App;
</code></pre>
```

### Code Block Transformations

By default, code blocks are processed for syntax highlighting only. To enable transformations (like TypeScript-to-JavaScript conversion), explicitly mark the code block:

```html
<pre><code class="language-typescript" data-transform="true">
const greeting: string = "Hello, world!";
console.log(greeting);
</code></pre>
```

This will generate both the original TypeScript and the transformed JavaScript versions.

## Output

After processing, the plugin:

1. **Replaces content** with a placeholder message
2. **Stores processed data** as JSON in the `data-precompute` attribute on the matched element (`<pre>`, `<section>`, or `<dl>`)
3. **Passes through metadata** like `data-name` and `data-slug` attributes
4. **Serializes custom props** in `data-content-props` attribute as JSON

```html
<pre
  data-precompute='{"Default":{"fileName":"index.ts","source":"...","transforms":{...}}}'
  data-name="My Demo"
  data-slug="my-demo"
  data-content-props='{"title":"Example Code"}'
>
  Error: expected pre tag with precomputed data to be handled by the CodeHighlighter component
</pre>
```

## Data Attributes

The following `data-*` attributes are supported on code elements:

### Reserved Attributes (handled internally)

| Attribute               | Purpose                                                                                                       |
| :---------------------- | :------------------------------------------------------------------------------------------------------------ |
| `data-filename`         | Sets the filename for the code variant                                                                        |
| `data-variant`          | Sets the variant name                                                                                         |
| `data-transform`        | Enables TypeScript→JavaScript transformation when `"true"`                                                    |
| `data-name`             | Demo name, passed through to the output `<pre>` element                                                       |
| `data-slug`             | URL slug for the demo, passed through to the output `<pre>` element                                           |
| `data-display-comments` | Keeps `@highlight`/`@focus` comments visible in the rendered source (set by the `displayComments` fence flag) |

The two output attributes, `data-precompute` and `data-content-props`, are also reserved — they are never forwarded as custom props.

### Custom Attributes (passed to content component)

Any other `data-*` attributes on the code element are serialized as JSON in the `data-content-props` attribute on the output `<pre>` element. These can be used by your content component for custom features.

**Example:**

```html
<pre><code class="language-typescript" data-title="Example">
const x = 1;
const y = 2;
</code></pre>
```

The output `<pre>` element will include:

```html
<pre data-content-props='{"title":"Example"}' ...></pre>
```

## Variant Naming

| Scenario                        | Variant Names                                                                     |
| :------------------------------ | :-------------------------------------------------------------------------------- |
| Single code block / lone `<dl>` | `"Default"`                                                                       |
| `<figcaption>` label            | The caption text with the ` variant` suffix stripped (e.g. `npm variant` → `npm`) |
| `data-variant` attribute        | Used when no `<figcaption>` label is present                                      |

## Supported Languages

| `class="language-*"` | Extension | Notes                                                                    |
| :------------------- | :-------- | :----------------------------------------------------------------------- |
| `javascript`         | `.js`     | Also accepts `js` (normalized)                                           |
| `typescript`         | `.ts`     | Also accepts `ts`; includes TS→JS transform when `data-transform="true"` |
| `tsx`                | `.tsx`    | React TypeScript                                                         |
| `jsx`                | `.jsx`    | React JavaScript                                                         |
| `json`               | `.json`   |                                                                          |
| `html`               | `.html`   |                                                                          |
| `css`                | `.css`    |                                                                          |
| `shell`              | `.sh`     | Also accepts `bash`, `sh`                                                |
| `yaml`               | `.yaml`   | Also accepts `yml`                                                       |
| `markdown`           | `.md`     | Also accepts `md`                                                        |
| Others               | `.txt`    | Fallback                                                                 |

## Integration with transformMarkdownCode

These plugins work together in a processing pipeline:

1. **[transformMarkdownCode](../transform-markdown-code/page.mdx)** (remark): Converts markdown code blocks with variants into semantic HTML
2. **transformHtmlCodeBlock** (rehype): Processes the HTML and precomputes data for rendering

### Complete Example

**Markdown Input:**

````markdown
npm

```bash variant-type=install
npm install package
```

pnpm

```bash variant-type=install
pnpm install package
```
````

**After transformMarkdownCode:**

```html
<section>
  <figure>
    <figcaption>npm variant</figcaption>
    <dl>
      <dd>
        <pre><code class="language-shell" data-variant="npm">npm install package</code></pre>
      </dd>
    </dl>
  </figure>
  <figure>
    <figcaption>pnpm variant</figcaption>
    <dl>
      <dd>
        <pre><code class="language-shell" data-variant="pnpm">pnpm install package</code></pre>
      </dd>
    </dl>
  </figure>
</section>
```

**After transformHtmlCodeBlock:**

```html
<section data-precompute='{"npm":{"fileName":"index.sh","source":"npm install package"...}}'>
  <!-- code contents replaced with a placeholder error message -->
</section>
```

## Performance

- Processes code blocks in parallel across the document
- Optimized text extraction with single-pass traversal
